# Partial Payments Design

**Date:** 2026-05-19
**Status:** Approved

## Problem

When a tenant pays less than the full monthly rent, the system has no way to record it cleanly:

- Marking the row as `collected` with a partial amount hides the shortfall from future months ‚Äî the `tenant_overdue` subquery excludes `collected` rows, so the 500 AED shortfall simply disappears.
- Leaving the row as `pending` loses the record that any money was received at all.
- There is no support for multiple top-up payments within the same month (e.g., 1,000 on May 5 and another 300 on May 12).

## Goals

- Record partial payments with full per-event detail (date, amount, method, receipt)
- Support multiple top-ups per month
- Carry the shortfall forward into `tenant_overdue` in future months
- Keep the current monthly-row data model intact (no structural overhaul)

---

## Data Model

### New table ‚Äî `payment_entries`

```sql
CREATE TABLE payment_entries (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  rent_payment_id INTEGER NOT NULL REFERENCES rent_payments(id),
  amount REAL NOT NULL,
  paid_date TEXT NOT NULL,
  payment_method TEXT CHECK(payment_method IN ('cash', 'cheque')),
  receipt_no TEXT,
  notes TEXT,
  recorded_by TEXT,
  recorded_at TEXT
);
CREATE INDEX idx_payment_entries_rent_payment ON payment_entries(rent_payment_id);
```

Each payment event (including top-ups) is a separate row. A month with two top-up payments has two `payment_entries` rows pointing at the same `rent_payments` row.

### Modified ‚Äî `rent_payments`

**Add `amount_paid` column:**
```sql
ALTER TABLE rent_payments ADD COLUMN amount_paid REAL NOT NULL DEFAULT 0;
UPDATE rent_payments SET amount_paid = amount WHERE status = 'collected';
```

`amount_paid` is a denormalized running total ‚Äî the sum of all `payment_entries` for this row. It is kept in sync whenever an entry is added or deleted. This avoids correlated subqueries into `payment_entries` in the main payments query.

**`amount` field:** Stays as the expected amount generated by the CTE (monthly or annual). It is **never overwritten** by payment operations going forward. The `PUT /rent-payments/:id` endpoint no longer accepts `amount` as an editable field.

**`status` gains a new value ‚Äî `partial`:** Some payment received but `amount_paid < expected_rent`. The DB CHECK constraint on `status` cannot be altered in SQLite, so it is enforced by Zod only. The existing `payment_method`, `paid_date`, `receipt_no`, `notes` columns on `rent_payments` are no longer written by payment operations (they live on entries now); they remain for backward compatibility with existing collected rows.

## Backend

### New route ‚Äî `POST /rent-payments/:id/entries`

**Request body:**
```typescript
{
  amount: number;           // positive
  paid_date: string;        // YYYY-MM-DD
  payment_method: 'cash' | 'cheque';
  receipt_no?: string;
  notes?: string;
}
```

**Handler logic:**
1. Insert row into `payment_entries`
2. Recompute `new_sum = SELECT SUM(amount) FROM payment_entries WHERE rent_payment_id = :id`
3. Fetch `expected_rent` for this payment (from joined contract)
4. Update `rent_payments`:
   - `amount_paid = new_sum`
   - `status = 'collected'` if `new_sum >= expected_rent`, else `status = 'partial'`
5. Write audit log: `payment.entry_added`
6. Return the inserted entry

### New route ‚Äî `DELETE /rent-payments/:id/entries/:entryId`

1. Delete the entry
2. Recompute `new_sum` (same as above; 0 if no entries remain)
3. Update `rent_payments`:
   - `amount_paid = new_sum`
   - `status = 'collected'` if `new_sum >= expected_rent`, else `'partial'` if `new_sum > 0`, else revert to `'pending'` (or `'overdue'` if `month < current_month`)
4. Write audit log: `payment.entry_deleted`

### New route ‚Äî `GET /rent-payments/:id/entries`

Returns all entries for the given `rent_payment_id`, ordered by `paid_date ASC`. Called when the user opens the payment popover.

### Modified ‚Äî `GET /rent-payments`

**`balance` expression** (simplified):
```sql
MAX(0, expected_rent - rp.amount_paid) as balance
```

**`tenant_overdue` subquery** ‚Äî partial rows contribute only their shortfall:
```sql
(SELECT COALESCE(SUM(
  CASE WHEN rp2.status = 'partial'
    THEN (CASE WHEN c2.payment_frequency = 'annual' THEN c2.annual_rent ELSE ROUND(c2.annual_rent / 12, 2) END - rp2.amount_paid)
    ELSE rp2.amount
  END
), 0)
FROM rent_payments rp2
JOIN contracts c2 ON rp2.contract_id = c2.id
WHERE c2.tenant_id = t.id
  AND rp2.status NOT IN ('collected')
  AND rp2.month < ?) as tenant_overdue
```

The response also includes `amount_paid` from the `rent_payments` row.

### Modified ‚Äî `GET /tenants`

`total_balance` subquery updated with the same partial-row shortfall logic:
```sql
(SELECT COALESCE(SUM(
  CASE WHEN rp.status = 'partial'
    THEN (CASE WHEN c2.payment_frequency = 'annual' THEN c2.annual_rent ELSE ROUND(c2.annual_rent / 12, 2) END - rp.amount_paid)
    ELSE rp.amount
  END
), 0)
FROM rent_payments rp
JOIN contracts c2 ON rp.contract_id = c2.id
WHERE c2.tenant_id = t.id
  AND rp.status NOT IN ('collected')) as total_balance
```

### Modified ‚Äî `PUT /rent-payments/:id`

Remove `amount`, `paid_date`, `payment_method`, `receipt_no` from the accepted Zod schema. Payment details are now entry-level. Keep `notes` (row-level annotation) and `status` (for manual overrides such as reverting to pending).

Also add `'partial'` to the status enum.

## Frontend

### New type ‚Äî `PaymentEntry`

```typescript
export type PaymentEntry = {
  id: number;
  rent_payment_id: number;
  amount: number;
  paid_date: string;
  payment_method: 'cash' | 'cheque' | null;
  receipt_no: string | null;
  notes: string | null;
  recorded_by: string | null;
  recorded_at: string | null;
};
```

### Modified type ‚Äî `RentPayment`

Add `amount_paid: number`.

### New hook ‚Äî `usePaymentEntries(rentPaymentId)`

`GET /rent-payments/:id/entries` ‚Äî enabled only when a payment ID is provided (i.e., the popover is open).

### New mutations ‚Äî `usePaymentEntryMutations`

- `addEntry(rentPaymentId, data)` ‚Üí `POST /rent-payments/:id/entries`, invalidates `['rent-payments']`
- `deleteEntry(rentPaymentId, entryId)` ‚Üí `DELETE /rent-payments/:id/entries/:entryId`, invalidates `['rent-payments']`

### Modified ‚Äî `CollectPopover` ‚Üí `PaymentPopover`

Renamed and repurposed. When opened, fetches entries for the payment. Shows two sections:

**Entry history** (only when entries exist):
```
May 5   Cash   1,000   Rcpt #101   [‚úï]
May 12  Cash     300   Rcpt #102   [‚úï]
‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
Total paid:  1,300 / 1,500
```
Each row has a delete button (‚úï). Deleting calls `deleteEntry` and re-fetches.

**Add payment form:**
- Amount field (pre-filled with remaining balance: `expected_rent - amount_paid`)
- Date field (defaults to today)
- Payment method toggle (Cash / Cheque), defaulting to contract's `payment_type`
- Receipt No. field
- Notes field
- "Record Payment" button ‚Üí calls `addEntry`

The trigger button (currently the status chip) retains the same appearance and position. The popover width expands slightly to accommodate the entry list (from `w-56` to `w-72`).

### Modified ‚Äî `PaymentsTab` table

**Status chip** ‚Äî add amber/orange style for `partial`:
```typescript
const STATUS_STYLE = {
  collected: 'bg-green-100 text-green-800 ...',
  partial:   'bg-orange-100 text-orange-800 ...', // new
  pending:   'bg-yellow-100 text-yellow-800 ...',
  overdue:   'bg-red-100 text-red-800 ...',
};
```

**Collected column** ‚Äî shows `amount_paid` for both `collected` and `partial` rows. For `partial`, appends a grey "/ {expected_rent}" so the shortfall is visible at a glance:
```tsx
{p.status === 'collected' && formatAED(p.amount_paid)}
{p.status === 'partial' && (
  <span>{formatAED(p.amount_paid)} <span className="text-muted-foreground">/ {formatAED(p.expected_rent)}</span></span>
)}
```

**`payment_method` chip** ‚Äî removed from the table row. Payment method detail is visible inside the popover's entry list. This avoids a dependency on the lazy-fetched entries data in the table render.

## Out of Scope

- Editing an existing entry (delete and re-add is the correction flow)
- Reporting by payment method or entry-level detail
- Partial payments on bill payments (separate system)
- Contract renewals (separate spec)
